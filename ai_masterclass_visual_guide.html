🤖 What is an LLM?
Large Language Model — the brain behind ChatGPT, Claude & Gemini
The original diagram — LLM definition, providers & API call flow

LLM = Language Model trained on a large corpus of data. A Language Model is an application that predicts the next word. That's it — no magic, just very clever probability maths!

Think of it like a super-powered autocomplete. When you type "Cat will sit on ___", the model reads everything before the blank and picks the most likely next word — "sofa". It uses a maths formula called P(next_word / context) — the probability of the next word given all the words before it. That's Bayes' theorem from school!

🏭

The Big 3 Providers

  • OpenAI — makes GPT-4, GPT-5. Powers ChatGPT. Best starting point for developers.
  • Gemini — Google's LLM. Free tier available via Google AI Studio.
  • Anthropic — makes Claude. Known for safety. Growing enterprise use.
🔌

How you talk to an LLM — via API

You (the developer) send a request to the LLM via an API call. The LLM sends a response back. You need VS Code + Python to write this code. No need to install the LLM — it runs on their servers, you just talk to it over the internet!

Cost fact: GPT-5 Mini = $0.25 per 1 MILLION input tokens. Your question has ~15 tokens. $5 gets you thousands of API calls! 💰

⚡ Transformers
The architecture inside every modern LLM — the black box revealed
User sends "I want to eat" → GPT processes → replies "I want to eat chocolate"

A Transformer is the neural network architecture that powers all modern LLMs (GPT, Claude, Gemini). Introduced in 2017 in the famous paper "Attention is All You Need." Before transformers, AI was much weaker. After? Everything changed.

When you send "I want to eat" to GPT, it goes through 4 steps internally before replying. Right now it looks like a black box — but the next 4 sections will open that box completely!

🎯

Why can't LLMs just read English directly?

Computers only understand numbers — not letters or words. The Transformer converts your English text into numbers, processes those numbers using layers of maths, and converts the output numbers back into English words. The 4 steps below explain exactly how!

✂️ Step 1 — Tokenization
Breaking your text into small numbered pieces the machine can work with
"I want to eat" → split into 4 individual tokens in boxes

A token is the smallest unit of text the LLM works with. Usually one word = one token. But longer words, hyphens, and special characters can split into multiple tokens. Each token gets a unique ID number from the LLM's vocabulary (its internal dictionary).

In the diagram, "I want to eat" becomes 4 separate boxes — one per word. The LLM doesn't see the whole sentence at once; it sees a sequence of numbered tokens.

🔢

Token IDs — the hidden numbers

Every token maps to a fixed number in that LLM's dictionary. OpenAI's numbers are different from Google's. Same word, different IDs between providers!

I
ID: 40
want
ID: 765
to
ID: 284
eat
ID: 4483
⚠️

Words don't always = 1 token

hands-on → 3 tokens: [hands] [-] [on]
eating → 1 token in modern models (older models: [eat] + [ing])
Each LLM has its own vocabulary — OpenAI has ~100,000 tokens in its dictionary!

04
📍 Step 2 — Static Embeddings
Converting token IDs into meaningful number coordinates in vector space
2D vector space — "I: 40" and "eating: 110" plotted as coloured dots

A static embedding converts each token ID into a long list of numbers called a vector. Think of it like GPS coordinates — but instead of 2 numbers (latitude, longitude), each word gets 1,536 numbers! Words with similar meanings get similar number patterns.

In the diagram, the 2D graph shows "I" and "eating" plotted at different positions. In reality, instead of 2 axes, we have 1,536 axes — impossible to draw, but the idea is identical. Each word occupies a unique position in this giant "meaning space."

eating (ID: 4483)
→
[0.12, -0.45, 0.87, 0.03,
-0.91, 0.22, 0.56, -0.34,
... 1,536 numbers total]
OpenAI standard = 1,536 dimensions. More = richer meaning!
🗺️

The "Meaning Map" analogy

"King" and "Queen" sit very close together in this 1,536D space because they have similar meanings. "King" and "Pizza" are very far apart. The machine can now tell which words are related just by measuring distances between their number-coordinates!

Static = the same word ALWAYS gets the same numbers, no matter the context. This causes a problem — solved in Step 3!
05
🧠 Step 3 — Contextual Embeddings
BERT models — understanding meaning from surrounding words bidirectionally
"Cat sat on mat" with arrows showing each word relates to all others. BERT Models. Output: [0.1, -0.99 ……… 1]

The word "fan" always gets the same static embedding numbers — but it means something totally different in these two sentences:
🔌 "Please turn on the fan" → electrical device
❤️ "I am your biggest fan" → admirer
Static embeddings can't tell the difference. Contextual embeddings fix this!

🔄

Bidirectional Context — the BERT revolution

In the diagram, arrows show "sat" connecting to every other word — Cat, on, mat. Each word updates its meaning based on ALL its neighbours (left AND right). That's "bidirectional." BERT = Bidirectional Encoder Representations from Transformers.

After processing, the entire sentence "Cat sat on mat" collapses into one big vector like [0.1, -0.99 … 1]. This is a sentence-level embedding — 1,536 numbers that capture the meaning of the WHOLE sentence in context!

🐦 BERT Models

Encoder-only. Read text in BOTH directions. Best for creating embeddings and understanding meaning. Used for RAG, search, and classification tasks.

🔮 GPT Models

Decoder-only. Read text left-to-right only (autoregressive). Used for TEXT GENERATION — your ChatGPT responses. All modern LLMs are GPT-style.

🎯 Step 4 — Predicting Tokens & The Token Cost Problem
One token at a time — and why uploading 1k+ page PDFs is a terrible idea
Top: LLM generates one token at a time until <end>. Bottom: User uploads 3 PDFs (1k+ pages each) — "How much sales we made in 2020?" This is the PROBLEM that RAG solves.
✦ Part A — How the LLM generates token by token ✦

🔄 The Generation Loop

The LLM predicts exactly ONE token per cycle. That token gets appended to the input. All 4 steps repeat. Another token. Repeat — until a special <end> token is predicted. This is why ChatGPT streams text word by word — you're watching each token appear live!

🌡️

Temperature

0 = always pick highest-probability token (boring but accurate). 1 = allow creative/risky tokens. ChatGPT's "creative" mode raises this!

💸 The Token Cost Problem

Look at the bottom of the diagram — a user uploads 3 PDFs with 1,000+ pages EACH, along with their question. Every page ~ 700 tokens. 3 PDFs × 1,000 pages × 700 tokens = 2.1 million tokens per query. At scale, that's insanely expensive!

🚨

The Real Problem

100 PDFs → ~70 MILLION tokens per question. No organisation would adopt that. This is exactly why we need Vector Databases + RAG — coming up in the next sections!

🗄️ Vector DB
Databases that store meaning (context) alongside content — the backbone of RAG
Postgres Database stores: Context [0.1,-0.99…1] → PDF, for 3 PDFs. Context column = vectors, Content column = original text.

A Vector Database stores data in two forms side-by-side:
Context — the embedding vector (the meaning in numbers)
Content — the original text chunk
When you search, you search by meaning — not exact keywords!

In the diagram, Postgres Database holds 3 rows. Each row has a yellow vector [0.1,-0.99…1] (the context/meaning) pointing to a PDF icon (the actual content). This is the "2 C's": Context + Content.

🐘

Why PostgreSQL?

Postgres + pgvector extension = store relational data AND vectors in ONE database. Data engineers love this — familiar SQL interface, no new tool to learn. You can even query vectors with SELECT statements!

🟢 ChromaDB

Free, open-source, runs locally. Perfect for development. Stores to disk so data survives restarts.

Hosted cloud vector DB. Production-ready. Scales to billions of vectors. Popular in enterprise.

📐 Semantic / Similarity Search
How the Vector DB finds the most relevant chunks using vector distances
Query vector for "How much sales we made in 2020" → compared in 2D space → Top 2 closest vectors retrieved from Vector DB (ranked 1 & 2)

Your user's query is converted into a vector. That query vector is then compared against ALL stored vectors in the database. The ones with the smallest distance (closest position in vector space) are declared the most relevant. Only the top-K are returned — you choose K (e.g., top 2, 3, or 5).

In the diagram, the query vector (purple) is plotted in 2D space. Two stored vectors (yellow and green) are very close to it — those are ranked 1 and 2. The distance formula used is 1 − cos(θ) — cosine similarity. Smaller value = more similar!

📏

Distance = Meaning Closeness

Query: "How much sales we made in 2020" → vector A
Stored chunk: "Q3 revenue report FY2020" → vector B
These two are very close in vector space → HIGH similarity → retrieved!
Chunk: "Employee birthday policy" → very FAR from the query → NOT retrieved.

# Retrieve top 3 most relevant chunks results = vector_db.similarity_search( "How much sales we made in 2020?", k=3 # Return top 3 matches ) # Each result has: metadata + page_content
You can set k=2, k=5, k=10 — more chunks = more context but more tokens. Balance is key! Typical production RAG uses k=3 to k=5.

The loop architecture that powers every modern AI agent
User → GPT → Loop ↔ Toolkit (SQL Tool, Google Tool, Search Tool) → Answer back. Bottom: create_agent — graph with start → llm → tool (dashed) → end

🏗️ Structured Output (Pydantic)
Forcing the LLM to return data in the exact schema your pipeline needs
User asks "What's the capital of France?" → GPT returns structured {Query:"...", Answer:"..."} — not free text!

Without structure, an LLM might answer "Paris!", "The capital is Paris.", or "Paris is the beautiful capital of France…" — all correct but IMPOSSIBLE to use in code that expects {query: "...", answer: "Paris"}. Unstructured output breaks downstream pipelines!

📐

Pydantic to the rescue!

Define a class inheriting from BaseModel. Specify exact field names and types. Pass to LLM via .with_structured_output(). The LLM is now FORCED to return data matching your schema — or throw a validation error!

from pydantic import BaseModel, Field class QueryResponse(BaseModel): query: str answer: str total_tokens: int = Field( description="Number of tokens used — must be integer!" ) # Now bind to LLM llm_structured = llm.with_structured_output(QueryResponse) result = llm_structured.invoke("Capital of France?") # Returns: QueryResponse(query="Capital of France?", answer="Paris", total_tokens=42)
Data engineers know: wrong data types break pipelines. Pydantic is your data contract between the LLM and everything downstream. Use it always in production! 🏭

⛓️ Templates & Chains (LCEL)
Reusable fill-in-the-blank prompts + pipeline steps connected with the | operator
Top: User sends Template{} → GPT returns {Query, Answer}. Bottom: Chains — one trigger runs all 5 steps in sequence automatically.

A Prompt Template is a reusable fill-in-the-blank prompt. Instead of hardcoding "Write a poem about mountains", you write "Write a poem about {topic}" — then swap in any topic at runtime. Write once, use thousands of times!

from langchain_core.prompts import ChatPromptTemplate template = ChatPromptTemplate.from_messages([ ("system", "You are a GenZ poet"), ("user", "Write a poem about {topic}") ]) # Reuse with ANY topic: t1 = template.invoke({"topic": "mountains"}) t2 = template.invoke({"topic": "ocean"})

A Chain connects multiple steps into one automated pipeline using the LCEL | pipe operator. Just like Linux pipes or dbt pipelines — each step's output feeds into the next step. Trigger once, all steps run!

# One trigger runs: Template → LLM → Parser chain = template | llm | output_parser # All 5 nodes in the bottom diagram fire automatically: result = chain.invoke({"topic": "mountains"})
🎯

Why data engineers love chains

You already build data pipelines — chains are AI pipelines! Raw data → transform → load. With chains: prompt → LLM → structured output → database. Same mental model, new domain. LCEL's | operator makes it beautifully readable.

A ReAct RAG Agent is a ReAct agent with your Vector DB added as one of the tools. So the LLM can now autonomously decide: "This question needs information from our internal documents" → calls the RAG tool → gets relevant chunks → answers! No manual retrieval needed.

🧩 Semantic Chunks
Smarter chunking — group sentences by meaning instead of character count
Document with 2 topic blocks (yellow + green). "Both are similar and the percentile is more than 70%" → they stay in the same chunk on the right.

Regular chunking: Split every 500 characters. Simple but can slice a sentence mid-thought — breaking meaning!

Semantic chunking: Compare each sentence to the next using embeddings. If similarity is >70%, keep them in the same chunk. If <70%, start a new chunk. Result: each chunk contains one complete idea.

In the diagram, the yellow block and green block inside the document represent two different topics. The yellow sentences are similar to each other (same chunk). The green sentences are a different topic (different chunk). The arrow shows: on the right, they become two separate, semantically pure chunks.

❌ Regular Chunk Risk

"The company was founded in 2016 by engineers
— / —
who believed in data." ← CUT HERE!

The sentence is broken. The chunk loses its meaning. RAG retrieves incomplete context!

✅ Semantic Chunk Win

"The company was founded in 2016 by engineers who believed in data. They started with 5 people."

Complete thought kept together! RAG retrieves full context → better answers.

from langchain_experimental.text_splitter import SemanticChunker from langchain_openai import OpenAIEmbeddings chunker = SemanticChunker( OpenAIEmbeddings(), breakpoint_threshold_type="percentile", breakpoint_threshold_amount=70 # 70% similarity threshold ) semantic_chunks = chunker.create_documents([full_text])

🧠 Memory in AI Agents
Making your agent remember past conversations — building your own lightweight ChatGPT
Rahul asks agent something → gets answer → memory stored in folder → next time Rahul asks "Could you please sum..." the agent has context from memory!

By default, every LLM call is completely stateless. If you tell it "My name is Dhruv" and immediately ask "What's my name?" in a NEW conversation — it has no idea. You must pass the conversation history yourself with every query. Memory = you managing that history list!

In the diagram, Rahul (user) asks the Agent something. The response is stored in a folder (memory). When Rahul's next message arrives ("Could you please sum…"), the stored history is automatically included — so the agent knows the full context of what was discussed before!

💾

3 Places to Store Memory

  • ⚡ RAM (Python list) — fastest, lost when program stops. For prototyping only.
  • 📁 Disk / JSON file — survives restarts. Good for single-user tools.
  • 🗄️ Database — production. One row per user per session. Scales to millions of users.
agent_memory = [] # RAM-based memory list def chat_with_memory(query: str): response = agent.invoke({ "messages": [{ "role": "user", "content": f"Answer: {query}. Previous context: {str(agent_memory)}" }] }) agent_memory.extend(response["messages"]) # Save for next time return response # Infinite chat loop — your own lightweight ChatGPT! while True: q = input("Ask me anything: ") print(chat_with_memory(q)) if input("Continue? yes/no: ") != "yes": break
