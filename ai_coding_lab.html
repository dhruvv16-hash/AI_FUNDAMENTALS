🧭 Text Embeddings
CH-1_LLMs / 2_Embeddings.ipynb — Converting text into 1536-dimensional numerical vectors
⚙️ Cell 1 — Create an Embedding Vector
from openai import OpenAI client = OpenAI() response = client.embeddings.create( input="I am a test string.", model="text-embedding-3-small" ) vector = response.data[0].embedding len(vector) # Output: 1536

What embeddings actually are

An embedding takes a piece of text and converts it into a list of 1536 floating-point numbers. Think of each number as a coordinate in a 1536-dimensional "meaning space". Similar sentences end up near each other; different sentences are far apart.

LineExplanation
client.embeddings.create(...)Calls OpenAI's embedding endpoint — completely separate from the chat/completion endpoint
input="I am a test string."The text to embed. Can be a single string or a list of strings for batch embedding
model="text-embedding-3-small"OpenAI's embedding model. "small" = 1536 dimensions, faster/cheaper. "large" = 3072 dimensions, more precise
response.data[0].embeddingThe response can embed multiple texts at once, so data is a list. [0] gets the first result, .embedding gets the actual vector list
len(vector) → 1536Confirms the vector has 1536 dimensions. Every string gets exactly 1536 numbers regardless of its length
🧭 Why 1536 numbers? Each dimension captures some latent feature of meaning (like "is this about finance?", "is this formal?", etc.). No single dimension has a human-readable meaning — they work together as a coordinate in meaning-space. "cat" and "kitten" will have very similar vectors; "cat" and "spaceship" will be very different.
⚙️ Cell 2 — Viewing the Raw Vector
vector # Prints: [0.00679779052734375, -0.019805908203125, -0.0287933349609375, ...] # 1536 numbers between roughly -0.12 and +0.12

What those numbers mean

The raw output is a Python list of 1536 floats, mostly between -1 and +1. Each number on its own means nothing — but comparing two vectors using cosine similarity tells you how semantically similar two texts are. This is the magic behind semantic search.

🔍 Cosine Similarity: If vectors A and B point in nearly the same direction, their cosine similarity is close to 1.0 (very similar). Perpendicular = 0.0 (unrelated). Opposite = -1.0 (antonyms). This number is how your vector database ranks results.

📝 STEP-1 — Creating Chunks of the text data
⚙️ Cell 2 — Recursive Character Text Splitter
splitter = RecursiveCharacterTextSplitter(chunk_size=100, chunk_overlap=20) chunks = splitter.split_documents(text_data) len(chunks) # Returns total number of chunks created

Understanding the splitter parameters

ParameterValueWhat it means
chunk_size=100100 charsEach chunk contains at most 100 characters. In production you'd use 500–2000 for richer context
chunk_overlap=2020 charsThe last 20 characters of chunk N are also the first 20 chars of chunk N+1. This prevents a sentence being split mid-thought and losing context at boundaries
✂️ Why "Recursive"? The splitter tries to break at the best boundary: first tries paragraph breaks (\n\n), then single newlines (\n), then spaces ( ), then individual characters as a last resort. This preserves natural language structure instead of cutting blindly at 100 chars.

split_documents(text_data) takes the full list of page Documents and returns a new (larger) list of smaller chunk Documents. Each chunk inherits the metadata from its parent page.

📝 STEP-2&3 — Creating Embeddings and Storing in Vector DB
⚙️ Cell 3 — ChromaDB Creation
from langchain_community.vectorstores import Chroma embed_model = OpenAIEmbeddings(model="text-embedding-3-small") chroma_db = Chroma.from_documents( chunks, embed_model, persist_directory="./chroma_db" )

Two steps happening in one call

CodeWhat happens
from langchain_community.vectorstores import ChromaImports LangChain's ChromaDB integration. ChromaDB is an open-source vector database that runs locally (no server needed)
OpenAIEmbeddings(model="text-embedding-3-small")Creates an embedding model object that can convert text → 1536-dim vectors. Stores the API key from environment
Chroma.from_documents(chunks, embed_model, ...)Does TWO things at once: (1) Calls embed_model to convert every chunk into a vector, (2) Stores all (vector, text, metadata) triples in ChromaDB on disk
persist_directory="./chroma_db"Saves the database to a folder called chroma_db. Without this, everything would be lost when Python stops
🗄️ What ChromaDB stores per chunk: The embedding vector (1536 floats), the text content, and metadata (source, page number). The folder on disk contains a SQLite file + binary index files for fast nearest-neighbor search.
📝 STEP-4 — Connection & Retrieval
⚙️ Cell 4 — Connect to Existing DB & Search
chroma_db_con = Chroma( persist_directory="./chroma_db", embedding_function=embed_model ) chroma_db_con.similarity_search( "By 2019, how many employees were there?", k=3 )

Re-connecting and querying

CodeExplanation
Chroma(persist_directory=..., embedding_function=...)Creates a connection to an already-existing ChromaDB. You must provide the embedding function again so it can embed your query before searching
similarity_search("query text", k=3)The full process: (1) Embeds "By 2019, how many employees..." into a vector, (2) Computes cosine similarity against all stored vectors, (3) Returns the top-3 most similar Document objects
k=3"k" means "top K". Returns exactly 3 chunks. Higher k = more context for LLM but also more tokens (and cost)
🔍 What you get back: A list of Document objects with their text and metadata. These chunks are the "relevant context" that will be passed to the LLM in Step 5.
📝 STEP-5 — LLM Integration and Answer Generation
⚙️ Cell 5 — Full RAG Q&A Loop
llm = ChatOpenAI(model="gpt-3.5-turbo", temperature=0) user_query = input("Enter your question: ") rel_chunks = chroma_db_con.similarity_search(user_query, k=3) rel_chunks_content = [] for i, chunk in enumerate(rel_chunks): rel_chunks_content.append(chunk.page_content) rel_chunks_content = str(rel_chunks_content) llm.invoke( f"{user_query}, Use the following context to answer the question: {rel_chunks_content}" )

The complete RAG flow in one cell

CodeExplanation
ChatOpenAI(model="gpt-3.5-turbo", temperature=0)Creates the answer-generation LLM. temperature=0 means deterministic — the model always picks the most probable next token. Higher temperature = more creative/random
input("Enter your question: ")Gets the user's question from the terminal at runtime
similarity_search(user_query, k=3)Embeds the user's question and finds the 3 most relevant chunks in ChromaDB
chunk.page_contentExtracts just the raw text from each Document object (without metadata)
str(rel_chunks_content)Converts the Python list to a string so it can be embedded inside the f-string prompt
llm.invoke(f"{user_query}...{rel_chunks_content}")Sends the combined prompt to GPT. The LLM sees both the question AND the relevant context, so it can answer from the document rather than hallucinating
💡 This is RAG in its purest form: "Here's the user's question. Here are the 3 most relevant passages from our document. Please answer the question using this context." The LLM is told to use the provided context rather than inventing facts.

🎯 Prompt Engineering
CH-3_Prompt_Engineering — Zero-shot, One-shot, Few-shot, Chain-of-Thought (theory + API calls)
⚙️ Setup — LangChain LLM
from langchain_openai import ChatOpenAI from dotenv import load_dotenv load_dotenv() llm = ChatOpenAI(model="gpt-3.5-turbo", temperature=0)

ChatOpenAI from LangChain wraps the OpenAI chat model. Now llm.invoke(prompt) sends any prompt and returns an AIMessage object with .content being the answer string.

📝 Zero-Shot Prompting
⚙️ Cell — Zero-Shot API Call
# Zero-shot: just the task, NO examples llm.invoke("Translate the following English sentence to French: 'How are you?'")

Zero-Shot: Rely on pre-trained knowledge

You give the LLM a task directly with no demonstrations. The model uses what it learned during training to complete the task. Best for simple, well-known tasks.

📋 Prompt structure: [Task instruction] + [Input]
Example: "Translate: [sentence]"
Works well when: The task is common and the model has seen many examples during training.
Risk: For ambiguous or complex tasks, the model might misunderstand the output format you want. That's why few-shot exists.
📝 One-Shot Prompting
⚙️ Cell — One-Shot API Call
one_shot_prompt = ( "Translate the following English sentence to French.\n" "Example: 'Good morning' -> 'Bonjour'\n" "Now, translate: 'How are you?'" ) llm.invoke(one_shot_prompt)

One-Shot: One example to set expectations

By providing one input→output example, you show the model the exact format you expect. The model picks up the pattern and applies it to the new input.

PartRole
"Translate..."Task instruction — tells the model what to do
"Example: 'Good morning' -> 'Bonjour'"The one demonstration — shows input → output format
"Now, translate: 'How are you?'"The actual task to complete
✅ Why Python string concatenation? Adjacent string literals in Python auto-concatenate. ("line 1\n" "line 2") is the same as "line 1\nline 2". This keeps long prompts readable.
📝 Few-Shot Prompting
⚙️ Cell — Few-Shot API Call
few_shot_prompt = ( "Translate the following English sentences to French.\n" "Example 1: 'Good morning' -> 'Bonjour'\n" "Example 2: 'Thank you' -> 'Merci'\n" "Now, translate: 'How are you?'" ) llm.invoke(few_shot_prompt)

Few-Shot: Multiple examples for complex tasks

Multiple demonstrations help the model identify patterns more reliably. Especially useful when: (1) The task has a specific output format, (2) The model might misinterpret the task, (3) You need consistent tone or structure.

📊 Trade-off: More examples = more tokens used = higher cost. But for structured extraction, classification, or formatting tasks, few-shot examples dramatically improve accuracy. Typical sweet spot: 3–10 examples.
📝 Chain-of-Thought Prompting
⚙️ Cell — CoT API Call
cot_prompt = ( "Solve the following math problem step by step.\n" "Question: If you have 10 apples and get 20 more, how many apples do you have?\n" "Example:\n" "Question: If you have 3 apples and get 2 more, how many apples do you have?\n" "Let's think step by step.\n" "Step 1: Start with 3 apples.\n" "Step 2: Get 2 more apples.\n" "Step 3: 3 + 2 = 5.\n" "Answer: 5 apples." ) llm.invoke(cot_prompt)

Chain-of-Thought: Force step-by-step reasoning

The magic phrase "Let's think step by step" unlocks LLM reasoning abilities. Without this, the model might jump to a wrong answer. With it, the model is forced to reason explicitly before answering.

🧠 Why does CoT work? When a model generates text token by token, each token is influenced by previous tokens. By forcing the model to write out reasoning steps, those intermediate steps become part of the context — and they anchor the final answer to be logically consistent with the reasoning.
PatternWhat it does
"Let's think step by step."The key trigger phrase — activates step-by-step reasoning mode
Full example with stepsShows the model exactly what "step by step" means for your specific task
"Answer: 5 apples."Shows the expected final format — the model will mirror this

The @tool decorator — most important concept in agentic AI

PartExplanation
from langchain.tools import toolImports the @tool decorator — LangChain's way to convert any Python function into an LLM-callable tool
@toolThis decorator wraps tool_duckduckgo_search so it has metadata the LLM can read. It registers the function name, parameters, and docstring
def tool_duckduckgo_search(query: str) -> str:Standard Python function with type hints. The type hints are required — they tell the LLM what types to pass
The docstring """Use this tool when..."""🚨 CRITICAL. This is what the LLM reads to decide WHEN to use the tool. Write it like instructions to the AI. Clear docstrings = better tool selection
DuckDuckGoSearchRun()A pre-built LangChain tool that queries DuckDuckGo search engine. No API key needed
search.invoke(query)Runs the actual web search and returns results as a string
tool_duckduckgo_search.invoke(...)After decoration, you can call the tool directly to test it. Returns a string of search results
⚙️ Cell 2 — Wikipedia Tool
@tool def tool_wikipedia_search(query: str) -> str: """Use this tool when you need to answer questions about persons, places, etc.""" from langchain_community.tools import WikipediaQueryRun from langchain_community.utilities import WikipediaAPIWrapper wikipedia = WikipediaQueryRun(api_wrapper=WikipediaAPIWrapper()) response = wikipedia.invoke(query) return response tool_wikipedia_search.invoke("Sam Altman")

Wikipedia tool breakdown

CodeExplanation
WikipediaAPIWrapper()Configures the Wikipedia API connection. Can set top_k_results and doc_content_chars_max to control how much text is retrieved
WikipediaQueryRun(api_wrapper=...)Creates the actual runnable tool using the wrapper. The wrapper handles rate limiting, API calls, and text extraction

⚙️ Cell 4 — Custom Personal Info Tool
@tool def tool_personal_info(name: str) -> str: """Use this tool when you need to answer questions about personal information. Args: name (str): The name of the person to look up. Returns: str: A string containing the person's age and occupation, or a message if not found. """ infos = [{ "name": "John Doe", "age": 30, "occupation": "Software Engineer" }, { "name": "Jane Smith", "age": 25, "occupation": "Data Scientist" }] for info in infos: if info["name"].lower() == name.lower(): return f"{info['name']} is {info['age']} years old and works as a {info['occupation']}." return "Information not found." tool_personal_info.invoke("John Doe") # → "John Doe is 30 years old and works as a Software Engineer."

Building a custom data lookup tool

This demonstrates how to wrap ANY Python logic as a tool. In production this would query a real database, CRM, or API — but the pattern is identical.

CodeExplanation
The Args / Returns docstringThe extended docstring format gives the LLM more detail about parameters. It knows name is a string person name, helping it pass correct values
infos = [{"name": ..., "age": ..., ...}]Simulated database — a hardcoded list of person records. In real projects this would be a database query
info["name"].lower() == name.lower()Case-insensitive comparison: "john doe" matches "John Doe". Robust to LLM capitalisation variations
return "Information not found."Graceful fallback when person isn't in the dataset. The LLM will see this string as the tool result and can explain it to the user
⚙️ Cell 5 — Bind All Tools to LLM
toolkit = [ tool_duckduckgo_search, tool_wikipedia_search, tool_arxiv_search, tool_personal_info ] llm_bind = llm.bind_tools(toolkit) llm_bind.invoke("What's the age of John Doe? Make tool calls if necessary.")

Tool binding — giving LLM the power to choose

CodeExplanation
toolkit = [...]A Python list of all tool functions (decorated with @tool). Order doesn't matter — the LLM reads all docstrings
llm.bind_tools(toolkit)Under the hood: converts each tool into an OpenAI function-calling schema (JSON with name, description, parameters) and sends it in every API request. Returns a new LLM object with tools attached
llm_bind.invoke("...")When invoked, the LLM can respond with either: (a) a text answer, OR (b) a tool_calls list specifying which tool to call and with what arguments
🤖 What happens under the hood: OpenAI's API receives the user message PLUS a list of "functions" (tool schemas). The model selects the best tool based on docstring matching, then returns a tool_calls field in the response instead of text. The caller is responsible for actually executing the tool and returning results. A ReAct agent automates this loop.

🔄 ReAct Agent
CH-4_Agent / 2_ReAct_Agent.ipynb — The Reason → Act → Observe loop that makes agents autonomous
⚙️ Cell — Create and Invoke a ReAct Agent
from langchain.agents import create_agent # Build the agent by combining the tool-aware LLM with the toolkit my_agent = create_agent(llm_bind, toolkit) # Invoke the agent with a user message my_agent.invoke( {"messages": [{"role": "user", "content": "What's the age of John Doe? Make tool calls if necessary."}]} )

The ReAct Loop — what happens when you invoke the agent

ReAct = Reasoning + Acting. The agent doesn't just call tools once — it loops until it has a final answer:

💭 Thought: Which tool do I need?
→
🔧 Action: Call tool_personal_info("John Doe")
→
👁️ Observe: "John Doe is 30 years old..."
→
💬 Final Answer to user
CodeExplanation
create_agent(llm_bind, toolkit)Creates a LangGraph-based agent that implements the ReAct loop. Takes the tool-bound LLM and the toolkit. Returns an agent you can invoke like any other chain
{"messages": [{"role": "user", "content": "..."}]}The agent takes its input as a dict with a messages key. This is OpenAI chat format — role is "user"/"assistant"/"system", content is the text
The agent return valueA dict with messages — a list of ALL messages including: user message, LLM reasoning, tool call requests, tool results, and the final answer
🔄 Multi-step example: If you ask "Who is Sam Altman and what papers has he published?", the agent will: (1) Call Wikipedia for his bio, (2) Call ArXiv for his papers, (3) Combine both into a final answer. All automatically, no code changes needed.

🏗️ Structured Output with Pydantic
CH-4_Agent / 3_Structured_Output.ipynb — Forcing the LLM to return typed, validated data
⚙️ Cell 1 — The Problem (Unstructured Output)
response = llm.invoke( "What is the capital of France? Generate output with: " "'Query: <the query>' 'Answer: <the answer>'" ) response = response.content response # → "Query: What is the capital of France? Answer: Paris" # This is just a string — fragile, hard to parse programmatically

Why unstructured output is a problem

When you ask an LLM to format output in a specific way, it sometimes adds extra text, changes the format, or makes mistakes. You can't reliably do response.split("Answer:")[1] because the LLM might write "The answer is:" or include extra sentences. Pydantic fixes this.

⚙️ Cell 2 — Pydantic Schema Definition
from pydantic import BaseModel, Field class llm_schema(BaseModel): query: str = Field(description="The query that was asked to the LLM") answer: str = Field(description="The answer generated by the LLM") total_tokens: int = Field(description="The total number of tokens used in the LLM response")

Defining the output schema

CodeExplanation
from pydantic import BaseModel, FieldPydantic is Python's most popular data validation library. LangChain uses it heavily for structured output
class llm_schema(BaseModel):Creates a data class (like a typed Python dict) that defines what fields the output must have
query: strThe query field must be a string. Pydantic auto-validates and raises an error if the LLM returns an int or list here
total_tokens: intThe total_tokens field must be an integer. This shows you can ask the LLM to count/estimate things as structured numbers
Field(description="...")The description tells the LLM exactly what to put in this field — it's sent to OpenAI's function-calling API as part of the schema. Clearer descriptions = better LLM compliance
⚙️ Cell 3 — with_structured_output & Invocation
llm_structured = llm.with_structured_output(llm_schema) response = llm_structured.invoke("What is the capital of France?") response # → llm_schema(query='What is the capital of France?', answer='Paris', total_tokens=42) response.model_dump() # → {"query": "What is the capital of France?", "answer": "Paris", "total_tokens": 42}

Structured output in action

CodeExplanation
llm.with_structured_output(llm_schema)Creates a new LLM wrapper that sends the Pydantic schema to OpenAI as a function-calling tool. The LLM is forced to return JSON matching that schema. Returns a modified LLM object
llm_structured.invoke("...")Calls the LLM and automatically parses the JSON response into an llm_schema Pydantic object. You get back a Python object with typed attributes, not a string
response.queryAccess fields like normal Python attributes: response.query, response.answer, response.total_tokens
response.model_dump()Converts the Pydantic object to a standard Python dictionary — useful for JSON serialization, database storage, or API responses
✅ Real-world use cases for structured output: (1) Extracting entities from text (name, date, amount as typed fields), (2) Classifying text into categories (must be one of: "positive", "negative", "neutral"), (3) Building APIs where downstream code expects predictable JSON, (4) Form filling from natural language.

📝 LCEL Chains — The Pipe Operator
⚙️ Cell 2 — Building a Full Chain
# Step-1: Create a prompt template my_template = ChatPromptTemplate.from_messages( [ ("system", "You are a genZ poet."), ("user", "Write a poem about the {topic}.") ] ) # Step-2: Structured output schema class llm_schema(BaseModel): query: str = Field(description="The query that was asked to the LLM") answer: str = Field(description="The answer generated by the LLM") total_tokens: int = Field(description="The total number of tokens used") llm_structured = llm.with_structured_output(llm_schema) # Step-3: Parser (optional with structured output) from langchain_core.output_parsers import PydanticOutputParser parser = PydanticOutputParser(pydantic_object=llm_schema) # Final Step: Chain everything with the pipe operator | my_chain = my_template | llm_structured my_chain.invoke({"topic": "beauty of mountains"})

LCEL — LangChain Expression Language

The | (pipe) operator creates a chain: the output of the left side automatically becomes the input of the right side. This is called LCEL (LangChain Expression Language).

CodeExplanation
my_chain = my_template | llm_structuredCreates a pipeline: template renders → output goes into LLM → LLM output is parsed into Pydantic schema. All with one |
my_chain.invoke({"topic": "beauty of mountains"})Runs the full pipeline: fills template → sends to LLM → gets structured object back. You never manually pass data between steps
PydanticOutputParserAn alternative parser for when you're not using with_structured_output. It parses LLM text output into a Pydantic object using regex/JSON parsing
⛓️ You can chain more steps: template | llm | parser | another_function — as many components as you want. Each component just needs to accept the output format of the previous one. This is how complex AI pipelines are built in production.

🧩 Semantic RAG (Smart Chunking)
CH-4_Agent / 5.1_RAG_Semantic.ipynb — Split by meaning, not character count, for better retrieval
⚙️ Cell 1 — Load & Join PDF Pages
docs = PyPDFLoader("../CH-2_RAG/NovaS.pdf").load() # Join all pages into one large string (SemanticChunker needs raw text) full_text = "\n".join([p.page_content for p in docs]) full_text

Regular RAG works on Document objects (one per page). SemanticChunker needs the entire document as a single string, so we join all pages with newlines. The list comprehension [p.page_content for p in docs] extracts text from each page Document, then "\n".join(...) merges them into one big string.

⚙️ Cell 2 — Semantic Chunker
from langchain_experimental.text_splitter import SemanticChunker chunker = SemanticChunker( OpenAIEmbeddings(model="text-embedding-3-small"), breakpoint_threshold_type="percentile", breakpoint_threshold_amount=60 ) semantic_chunks = chunker.create_documents([full_text]) semantic_chunks

How SemanticChunker works — sentence by sentence

ParameterExplanation
OpenAIEmbeddings(model="text-embedding-3-small")The embedding model used to compare sentences. Each sentence gets embedded and compared to its neighbors
breakpoint_threshold_type="percentile"Uses the statistical percentile of cosine distances as the threshold. Alternative: "standard_deviation", "interquartile"
breakpoint_threshold_amount=60Creates a new chunk when the distance between consecutive sentences is in the 60th percentile or above (i.e., when sentences become notably less similar). Lower % = more chunks (splits more aggressively)
🧩 The algorithm step-by-step:
1. Split text into sentences
2. Embed every sentence
3. For each pair of adjacent sentences, compute cosine distance (how different are they?)
4. Find the 60th percentile distance threshold
5. Wherever distance exceeds threshold = topic change → start new chunk
6. Result: chunks that contain complete ideas, not arbitrary 500-char slices

chunker.create_documents([full_text]) takes the text as a list (can process multiple texts) and returns a list of Document objects — same format as regular chunking.

⚙️ Cell 3–5 — Rest of Pipeline (Same as Regular RAG)
# Store in ChromaDB (DIFFERENT folder: chroma_db_semantic) chroma_db = Chroma.from_documents( semantic_chunks, embed_model, persist_directory="./chroma_db_semantic" ) # Connect and search chroma_db_con = Chroma(persist_directory="./chroma_db_semantic", embedding_function=embed_model) chroma_db_con.similarity_search("By 2019, how many employees were there?", k=3) # Answer generation (identical to regular RAG) llm = ChatOpenAI(model="gpt-3.5-turbo", temperature=0) user_query = input("Enter your question: ") rel_chunks = chroma_db_con.similarity_search(user_query, k=3) rel_chunks_content = str([chunk.page_content for chunk in rel_chunks]) llm.invoke(f"{user_query}, Use the following context to answer the question: {rel_chunks_content}")

Everything after chunking is identical to regular RAG. The only differences are: (1) chroma_db_semantic folder name to keep databases separate, (2) The chunk quality is better because they contain complete ideas. The retrieval and LLM call are the same API calls.

⚠️ Cost note: SemanticChunker calls the embedding API once for every sentence in your document during chunking — this costs money. Regular chunking is free (no API calls during chunking). For large documents, semantic chunking can be significantly more expensive to index.

🤖 ReAct RAG Agent
CH-4_Agent / 5.2_RAG_Agent.ipynb — RAG becomes a tool, giving the agent access to private documents
⚙️ The Key Addition — tool_rag Function
@tool def tool_rag(query: str) -> str: """Use this tool when you need to answer questions based on NovaSphere Organization's documentation.""" from langchain_community.vectorstores import Chroma from langchain_openai import OpenAIEmbeddings embed_model = OpenAIEmbeddings(model="text-embedding-3-small") chroma_db_con = Chroma( persist_directory="./chroma_db_semantic", embedding_function=embed_model ) # Retrieve relevant documents from the vector store relevant_docs = chroma_db_con.similarity_search(query, k=2) relevant_docs_content = "\n".join([doc.page_content for doc in relevant_docs]) return relevant_docs_content tool_rag.invoke("When was NovaSphere Organization founded?")

Wrapping the entire RAG pipeline as a single tool

This is the key insight: the whole retrieval pipeline (embed query → vector search → return text) becomes ONE tool that the agent can choose to call.

CodeExplanation
The docstring specifies "NovaSphere Organization's documentation"The LLM will only call this tool when asked about NovaSphere. For general questions, it will use DuckDuckGo or Wikipedia instead. The docstring = routing logic
Imports inside the function bodyImports are placed inside the function so they're loaded on-demand. Good practice for optional dependencies
similarity_search(query, k=2)Returns only 2 chunks (instead of 3 in regular RAG) — agents have token limits since they accumulate many messages in their context
"\n".join([doc.page_content for doc in relevant_docs])Joins the 2 retrieved chunks into one string with newlines. This string is returned to the agent as the "tool result" — it becomes an Observation in the ReAct loop
⚙️ Full Toolkit + Agent with RAG
toolkit = [ tool_duckduckgo_search, tool_wikipedia_search, tool_arxiv_search, tool_personal_info, tool_rag # ← NEW: private document access ] llm_bind = llm.bind_tools(toolkit) my_agent = create_agent(llm_bind, toolkit) # Test 1: Uses tool_personal_info my_agent.invoke( {"messages": [{"role": "user", "content": "What's the age of John Doe?"}]} ) # Test 2: Uses tool_rag (private docs) my_agent.invoke( {"messages": [{"role": "user", "content": "When was NovaSphere Organization founded?"}]} )

The agent now autonomously routes between 5 tools. For "When was NovaSphere founded?" → the agent sees the docstring says "NovaSphere documentation" → calls tool_rag → gets the answer from the PDF. For John Doe → uses tool_personal_info. No hardcoding. The docstrings do all the routing.

🧠 Agent Memory
CH-4_Agent / 6_Agent_Memory.ipynb — Persistent conversation history so the agent remembers you
⚙️ Cell 1 — The Memory Container
# Simple Python list = RAM-based memory agent_memory = []

Memory = just a Python list

There's no magic here. Memory is literally a Python list that accumulates LangChain message objects. Each time the agent responds, all messages (user, AI, tool calls, tool results) get appended to this list. The list is then included in future prompts so the agent has conversation history.

📦 Types of memory storage:
• RAM (this code): Fastest, lost when Python stops. Good for single-session prototypes
• JSON/file: Survives restarts, easy to inspect. Good for single-user tools
• Database (PostgreSQL/Redis): Production-grade, supports millions of users with separate conversation histories per user
⚙️ Cell 2 — The Memory-Aware Invoke Function
def invoke_agent_with_memory(query: str): response = my_agent.invoke( {"messages": [{ "role": "user", "content": f"Answer this query: {query}. You can refer to the previous interactions: {str(agent_memory)}" }]} ) agent_memory.extend(response['messages']) return response

Memory function — explained line by line

LineExplanation
f"Answer this query: {query}. You can refer to the previous interactions: {str(agent_memory)}"Injects the entire conversation history into the prompt. The LLM has no persistent state between calls — you must explicitly include history in every new call. str(agent_memory) converts the list of message objects to a readable string
agent_memory.extend(response['messages'])After the agent responds, appends ALL generated messages to memory. extend adds multiple items (unlike append which adds one). The response contains: user message, LLM reasoning, any tool calls, tool results, and final answer
return responseReturns the full response dict so the caller can print or process it
⚠️ Memory grows with every turn! After 10 conversations, str(agent_memory) could be thousands of characters. This increases cost and can hit token limits. Production solutions: truncate old messages, summarize history, or use a dedicated memory store like LangGraph's checkpointing.
